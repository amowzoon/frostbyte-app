# FrostByte Alert App

A React Native mobile application for the FrostByte black ice detection system. The app provides real-time black ice alerts to pedestrians and drivers based on detections from FrostByte sensor units deployed in the field.

---

## Related Repositories and Branches

| Repository | Branch | Description |
|---|---|---|
| `frostbyte-app` | `main` | This repository — the React Native alert app |
| `frostbyte-backend` | `test-py-files` | Backend server — modified to support the alert app |
| `frostbyte` | `test-py-files` | On-device Pi pipeline — no changes made |

---

## Overview

FrostByte is a senior design project (EC463) at Boston University that uses a Raspberry Pi 5 with an RGB camera (IMX708), IR camera (FLIR Lepton), mmWave radar (TI IWR6843), and temperature sensors to detect black ice on road surfaces. This app is the consumer-facing component of the system — it receives alerts generated by the backend inference pipeline and displays them on a live map centered on the user's current location.

---

## Features

- Live ice alert map displaying nearby black ice detections as color-coded markers (red = high confidence, orange = medium, yellow = low)
- 30-second polling for new alerts within 1km of the user's location
- Push notifications when high-confidence ice is detected nearby via Expo push infrastructure
- Email and password authentication with JWT-based session management persisted across app restarts
- Guest mode allowing map and alert access without creating an account
- Configurable alert radius per user account (100m to 2km)
- 8-second location timeout with last-known-position fallback so the map loads even if GPS is slow

## Setup and Running

### Prerequisites

- Node.js v20. Node v24 breaks the Metro bundler — use nvm-windows to manage versions on Windows.
- Expo Go installed on your phone from the App Store or Google Play.
- FrostByte backend running and reachable on your local network (frostbyte-backend, test-py-files branch).

### Install dependencies

```bash
cd frostbyte-app
npx expo install expo-location expo-notifications react-native-maps axios
npx expo install "@react-navigation/native" "@react-navigation/stack@^6"
npx expo install react-native-screens react-native-safe-area-context react-native-gesture-handler
npx expo install @react-native-community/slider @react-native-async-storage/async-storage
```

### Configure backend URL

Open `api/client.js` and set `BASE_URL` to your computer's local IPv4 address:

```javascript
export const BASE_URL = 'http://YOUR_LOCAL_IP:8000';
```

Find your IP by running `ipconfig` on Windows and looking for the Wi-Fi adapter's IPv4 Address. Your phone must be on the same Wi-Fi network as the computer running the backend.

### Start the backend

```bash
cd frostbyte-backend
docker compose up
```

Wait for `Application startup complete` in the logs before launching the app.

### Run the app

```bash
cd frostbyte-app
npx expo start
```

Scan the QR code with Expo Go on your phone.

Note: This project targets Expo SDK 54. Use `--template blank@sdk-54` when recreating the project to ensure compatibility with the App Store version of Expo Go. SDK 55 is currently in beta and requires TestFlight on iOS physical devices.

## Debugging

### Check backend health

Before launching the app, verify the backend is reachable by opening this URL in a browser on your computer:

```
http://10.0.0.18:8000/health
```

This should return `{"status": "ok"}`. If the page does not load, Docker is not running or the backend has crashed. Check `docker compose logs server` for the error.

### Test API endpoints directly

The backend exposes a full interactive API reference at:

```
http://10.0.0.18:8000/docs
```

This is the primary debugging tool for the alert app endpoints. Each endpoint can be tested directly from the browser without needing the phone. To test account registration specifically:

```
http://10.0.0.18:8000/docs#/alert-app/register_api_app_register_post
```

Click the endpoint, click Try it out, enter an email and password, and click Execute. The response body and status code will show exactly what the server returned. This is how the bcrypt incompatibility was discovered during development — the /docs interface returned a 500 with no detail, while `docker compose logs server` revealed the full Python traceback pointing to `alert_api.py` line 137.

### View server logs

To see live server errors without stopping the backend, open a second terminal window:

```bash
cd frostbyte-backend
docker compose logs server
```

This prints the full log output including Python tracebacks from any failed requests. Check this whenever the app returns an unexpected error.

### Common issues

**Registration failed or something went wrong** — open the /docs interface and test the register endpoint directly. The response body will show the exact server error. Then check `docker compose logs server` for the Python traceback.

**App stuck on getting your location** — check that Location Services are enabled on your phone under Settings > Privacy and Security > Location Services > Expo Go, set to While Using the App. If permissions are correct, shake the phone to open the Expo developer menu and tap Reload.

**Connection refused when accessing 10.0.0.18:8000** — Docker Desktop is not running, or the backend container has crashed. Open Docker Desktop first, then run `docker compose up` in the frostbyte-backend directory.

**Project is incompatible with this version of Expo Go** — the project SDK version does not match Expo Go. Recreate the project using `--template blank@sdk-54` to pin to SDK 54, which is the version currently supported by the App Store release of Expo Go.

**Metro bundler error on npx expo start** — you are running Node v24. Switch to Node v20 using `nvm use 20`, delete node_modules, then reinstall with `npm install --legacy-peer-deps`.

## Backend Changes (frostbyte-backend, test-py-files branch)

The following changes were made to frostbyte-backend to support the alert app. All changes are backward compatible and do not affect the existing labeling tool, capture pipeline, or dashboard.

### New file: server/alert_api.py

A new FastAPI router mounted at `/api/app` that provides all app-facing endpoints:

| Endpoint | Method | Auth | Description |
|---|---|---|---|
| `/api/app/register` | POST | None | Create account, returns JWT |
| `/api/app/login` | POST | None | Login, returns JWT |
| `/api/app/push-token` | POST | JWT | Register Expo push token |
| `/api/app/settings` | GET | JWT | Get alert radius and email |
| `/api/app/settings` | PATCH | JWT | Update alert radius |
| `/api/app/alerts/nearby` | GET | JWT | Active ice alerts within radius of GPS coordinates |
| `/api/app/alerts` | POST | None | Internal endpoint called by inference pipeline after detection |

Authentication uses JWT tokens signed with `ALERT_JWT_SECRET` (configurable via environment variable, defaults to a placeholder that should be changed in production). Passwords are hashed using `bcrypt` directly via `bcrypt.hashpw` and `bcrypt.checkpw`.

### Modified file: server/main.py

Three additions:

1. `from alert_api import alert_router` added to the imports alongside the existing routers.
2. `app.include_router(alert_router)` added after the existing `include_router` calls.
3. Six table and column creation statements added inside `startup()`, all using `IF NOT EXISTS` so they are safe to run on an existing database:
   - `latitude` and `longitude` columns added to the existing `devices` table so the Pi can report its GPS position on registration.
   - `app_users` table storing registered app users: email, bcrypt password hash, Expo push token, alert radius preference, and last seen timestamp.
   - `ice_alerts` table storing detected ice events with GPS coordinates, confidence score, session reference, and expiry timestamp.
   - Spatial indexes on both tables for fast bounding box proximity queries.

### Modified file: server/processing_api.py

Two changes:

1. The `mobile_sam` import block at the top of the file is wrapped in a `try/except` with a `print()` statement (not `log`, which is not yet defined at module load time) so the server starts cleanly even when MobileSAM model files are not present in the container. Sets `ROAD_ISOLATION_AVAILABLE = False` when unavailable.
2. At the end of `run_inference()`, after the confidence map is computed, a block fires a POST to `/api/app/alerts` when `mean_confidence > 0.5`. The device GPS coordinates are pulled from the `devices` table using the session's `device_id`. The existing return dictionary gains one new field: `alert_fired` (boolean).

### Modified file: server/requirements.txt

Two packages added:

```
bcrypt
python-jose[cryptography]
```

`passlib[bcrypt]` was explicitly avoided. The version of `bcrypt` installed in the container is incompatible with passlib's internal version detection, producing a `ValueError` on every hash attempt regardless of password length. The `bcrypt` package is used directly instead.

## Known Limitations

- The `/api/app/alerts` internal endpoint has no authentication guard. In a production deployment this should be protected with an internal API key to prevent unauthorized alert injection.
- Push notification proximity uses a fixed 1km server-side radius. The per-user alert radius preference controls what is displayed on the map but does not yet filter push notifications.
- Guest mode provides full map access but cannot receive push notifications or persist settings since no account exists.
- Expo push notifications require a physical device and will not fire in the iOS simulator.
- The app polls for alerts every 30 seconds. There is no WebSocket connection — this is intentional to keep the server stateless and scalable.
